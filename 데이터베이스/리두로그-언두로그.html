REDO &
UNDO LOG

트랜잭션의 내구성과 원자성을 보장하는 두 개의 로그.
DB가 어떤 상황에서도 데이터를 잃지 않는 비밀이 여기에 있다.

01 WHY LOGS

왜 트랜잭션 로그가 필요한가

데이터베이스는 성능을 위해 변경 사항을 메모리(Buffer Pool)에 먼저 반영한다. 하지만 메모리는 휘발성이다.

문제: 메모리는 휘발성

COMMIT 직후 서버가 갑자기 죽으면?
Buffer Pool에만 있던 변경 사항은 사라진다.

해결: 로그로 보호

변경 내역을 디스크의 로그 파일에 먼저 기록해 두면,
장애 후에도 데이터를 복구할 수 있다.

ACID와 로그의 관계

ACID 속성보장 수단관련 로그
Atomicity 트랜잭션은 전부 반영되거나 전부 취소된다 Undo Log
Consistency 제약 조건 위반 시 롤백으로 되돌린다 Undo Log
Isolation 다른 트랜잭션의 미완료 변경을 보지 않는다 Undo Log (MVCC)
Durability COMMIT된 데이터는 장애에도 유지된다 Redo Log (WAL)
핵심: Redo Log는 Durability를 보장하고, Undo Log는 Atomicity + Isolation을 보장한다. 두 로그가 협력해야 완전한 ACID가 완성된다.
02 REDO LOG

Redo Log — 변경 후의 값을 기록한다

Redo Log는 "이 페이지를 이렇게 바꿨다"는 변경 결과(after-image)를 기록하는 로그다. 장애 후 재실행(Redo)에 사용된다.

WAL (Write-Ahead Logging): 데이터 파일을 수정하기 전에 반드시 로그를 먼저 디스크에 기록한다. 이 원칙 덕분에 데이터 파일이 깨져도 로그만 있으면 복구가 가능하다.

Redo Log 기록 형식

LSN 101 BEGIN Tx#42 트랜잭션 시작
LSN 102 WRITE Page 5, Off 120 → name = 'Kim'
LSN 103 WRITE Page 8, Off 64 → balance = 3000
LSN 104 COMMIT Tx#42 커밋 완료

Sequential Write

Redo Log는 항상 끝에 이어서 쓴다(Append-Only). 디스크 Seek 없이 Sequential I/O로 동작하므로 매우 빠르다.

COMMIT = Log Flush

COMMIT 시점에 Redo Log를 디스크에 fsync한다. 데이터 파일 자체는 아직 안 바꿔도 된다. 이것이 WAL의 핵심이다.

InnoDB Redo Log 구조

Redo Log Buffer 메모리
→ fsync →
ib_logfile0 디스크
→
ib_logfile1 디스크
→ 순환 →
Checkpoint Data File 반영

Redo Log는 순환 버퍼(Circular Buffer) 구조다. Checkpoint가 완료된 영역은 재사용된다.

주의: Redo Log 파일 크기가 너무 작으면 Checkpoint가 자주 발생하여 성능이 떨어진다. 반대로 너무 크면 장애 복구 시간이 길어진다.
03 UNDO LOG

Undo Log — 변경 전의 값을 기록한다

Undo Log는 "이 페이지가 원래 이랬다"는 원본 값(before-image)을 기록하는 로그다. 롤백과 MVCC에 사용된다.

Undo Log 기록 형식

Undo#1 INSERT Page 5 → row PK=7 삽입됨 (삭제하면 롤백)
Undo#2 UPDATE Page 5, Off 120 ← name = 'Lee' (원래 값)
Undo#3 UPDATE Page 8, Off 64 ← balance = 5000 (원래 값)

Rollback 지원

ROLLBACK 실행 시 Undo Log를 역순으로 읽어 원래 값으로 되돌린다. Atomicity의 핵심 메커니즘이다.

MVCC 지원

다른 트랜잭션이 수정 중인 데이터를 읽을 때, Undo Log에서 해당 시점의 과거 버전을 제공한다.

Redo Log와 구조적 차이

항목Redo LogUndo Log
기록 내용 변경 후의 값 (after-image) 변경 전의 값 (before-image)
쓰기 방식 Sequential (Append-Only) 페이지 내 제자리 쓰기
저장 위치 별도 로그 파일 (ib_logfile) Undo Tablespace (시스템 또는 별도)
수명 Checkpoint 완료 시 재사용 관련 트랜잭션이 모두 종료되면 Purge
주 역할 Crash Recovery (재실행) Rollback + MVCC
기억법: Redo = "다시 하다" → 커밋된 변경을 재실행. Undo = "되돌리다" → 미완료 변경을 취소.
04 WRITE FLOW

트랜잭션 쓰기 흐름 전체 과정

UPDATE 문 하나가 실행될 때 InnoDB 내부에서 일어나는 일을 순서대로 따라가 보자.

1

Buffer Pool에서 페이지 읽기

대상 페이지가 Buffer Pool에 없으면 디스크에서 읽어온다. 이후 모든 작업은 메모리에서 수행된다.

2

Undo Log에 원래 값 저장

수정하기 전에 현재 값(before-image)을 Undo Log에 기록한다. 롤백과 MVCC를 위한 보험이다.

3

Buffer Pool 페이지 수정 (Dirty Page)

메모리 상의 페이지를 직접 수정한다. 이 시점에서 페이지는 "Dirty" 상태가 된다.

4

Redo Log Buffer에 변경 기록

변경 후의 값(after-image)을 Redo Log Buffer에 기록한다. 아직 디스크에 쓰지 않은 상태다.

5

COMMIT → Redo Log Flush (fsync)

Redo Log Buffer를 디스크의 Redo Log 파일에 강제 기록한다. 이 시점에서 트랜잭션의 Durability가 보장된다.

6

나중에: Checkpoint가 Dirty Page를 Data File에 반영

Background 스레드가 주기적으로 Dirty Page를 디스크의 데이터 파일에 쓴다. COMMIT과는 비동기적으로 수행된다.

핵심 포인트: COMMIT은 데이터 파일을 바꾸는 게 아니라 Redo Log를 디스크에 쓰는 것이다. 실제 데이터 파일 반영은 Checkpoint가 나중에 처리한다. 이것이 WAL의 본질이다.
05 CRASH RECOVERY

장애 복구: Redo 그리고 Undo

서버가 갑자기 죽었을 때, 두 로그를 사용해 정확히 커밋된 상태로 되돌리는 과정이다.

장애 시점의 상태

Tx#1 — COMMIT 완료: Redo Log에 기록됨, 하지만 Data File에는 미반영
Tx#2 — 진행 중: Buffer Pool에 수정 있음, COMMIT 전에 장애 발생
Tx#3 — COMMIT 완료: Redo Log 기록됨, Data File에도 이미 반영됨

복구 목표

Tx#1: COMMIT됐으니 변경을 재실행(Redo)
Tx#2: COMMIT 안 됐으니 변경을 취소(Undo)
Tx#3: 이미 반영됐으니 아무것도 안 함

복구 과정 (2단계)

Phase 1: Redo (Roll Forward)

마지막 Checkpoint 찾기

Redo Log에서 마지막 Checkpoint LSN을 확인한다.

Checkpoint 이후 로그 전부 재실행

커밋 여부와 관계없이 모든 변경을 순서대로 다시 적용한다.

Buffer Pool 상태 복원 완료

장애 직전의 메모리 상태가 그대로 복원된다.

Phase 2: Undo (Roll Back)

미완료 트랜잭션 식별

COMMIT 레코드가 없는 트랜잭션을 찾는다.

Undo Log로 변경 취소

미완료 트랜잭션의 변경 사항을 역순으로 되돌린다.

일관된 상태 도달

COMMIT된 것만 남고, 미완료는 모두 사라진다.

"Redo는 있어야 할 것을 되살리고, Undo는 없어야 할 것을 지운다."
이 두 단계를 거치면 DB는 장애 직전에 커밋된 정확한 상태로 돌아간다.
06 MVCC

Undo Log와 MVCC

MVCC(Multi-Version Concurrency Control)는 읽기 트랜잭션이 쓰기 트랜잭션을 기다리지 않게 해준다. 그 비밀은 Undo Log의 버전 체인이다.

버전 체인 (Undo Log Chain)

하나의 행이 여러 트랜잭션에 의해 수정되면, Undo Log에 이전 버전들이 체인으로 연결된다.

현재 값 name = 'Park'
Tx#50 수정
← undo
이전 버전 name = 'Kim'
Tx#45 수정
← undo
최초 버전 name = 'Lee'
Tx#30 생성

읽기 트랜잭션의 동작

1

트랜잭션 시작 시 Read View(스냅샷)를 생성한다

2

행의 현재 버전이 내 Read View보다 새로우면 건너뜀

3

Undo Log 체인을 따라가며 내 시점의 버전을 찾는다

4

잠금 없이 일관된 읽기(Consistent Read) 완료

REPEATABLE READ: 트랜잭션 시작 시점의 스냅샷을 계속 사용한다. 같은 쿼리를 여러 번 실행해도 동일한 결과를 보장한다.
READ COMMITTED: 각 쿼리 실행 시점마다 새 스냅샷을 생성한다. 같은 트랜잭션 내에서도 다른 결과가 나올 수 있다.
Purge Thread: 어떤 트랜잭션도 더 이상 참조하지 않는 오래된 Undo Log 버전은 Purge Thread가 정리한다.
07 PRACTICE

InnoDB 실무 가이드

Redo Log와 Undo Log 관련 핵심 설정과 모니터링 포인트를 정리한다.

Redo Log 설정

# Redo Log 전체 크기 (MySQL 8.0.30+)
innodb_redo_log_capacity = 2G

# fsync 전략
innodb_flush_log_at_trx_commit = 1
# 1: 매 COMMIT마다 fsync (안전)
# 2: OS 캐시까지만 (빠름, 약간 위험)
# 0: 1초마다 flush (가장 빠름, 위험)

권장: 데이터 무결성이 중요하면 반드시 1을 사용한다. 2나 0은 최대 1초 분량의 트랜잭션을 잃을 수 있다.

Undo Log 설정

# Undo Tablespace 자동 생성 (8.0+)
innodb_undo_tablespaces = 2

# Undo Log 최대 크기
innodb_max_undo_log_size = 1G

# 자동 Truncate 활성화
innodb_undo_log_truncate = ON

주의: 장시간 트랜잭션은 Undo Log가 정리되지 못하게 하여 디스크를 가득 채울 수 있다.

모니터링 쿼리

-- Redo Log 상태 확인
SHOW ENGINE INNODB STATUS\G
-- LOG 섹션에서 LSN, Checkpoint 위치 확인
-- Log sequence number: 현재 LSN
-- Last checkpoint at: 마지막 체크포인트 LSN
-- 차이가 크면 Checkpoint가 밀리고 있는 것

-- Undo Log (History List Length) 확인
SHOW ENGINE INNODB STATUS\G
-- TRANSACTIONS 섹션의 History list length
-- 이 값이 계속 증가하면 Purge가 밀리고 있는 것

-- 장시간 트랜잭션 찾기
SELECT trx_id, trx_state, trx_started,
       TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration_sec
FROM information_schema.innodb_trx
ORDER BY trx_started ASC;

장시간 트랜잭션

열어놓고 안 닫는 트랜잭션은 Undo Log Purge를 막아 디스크 폭증을 유발한다. History List Length를 모니터링하라.

Redo Log 크기 부족

Redo Log가 작으면 급격한 Checkpoint(Async/Sync Flush)가 발생해 순간적으로 쓰기가 멈출 수 있다.

flush_log 설정 변경

성능을 위해 innodb_flush_log_at_trx_commit을 0이나 2로 바꾸면, 장애 시 커밋된 데이터를 잃을 수 있다.

08 SUMMARY

최종 정리

Redo Log

변경 후의 값(after-image)을 기록한다
WAL 원칙에 따라 데이터보다 먼저 디스크에 쓴다
Sequential Write로 성능이 뛰어나다
COMMIT = Redo Log를 fsync하는 것이다
장애 복구 시 커밋된 트랜잭션을 재실행한다

Undo Log

변경 전의 값(before-image)을 기록한다
ROLLBACK 시 원래 값으로 되돌린다
버전 체인으로 MVCC를 지원한다
Purge Thread가 불필요한 버전을 정리한다
장애 복구 시 미완료 트랜잭션을 취소한다
관점Redo LogUndo Log
한마디 정의 "이렇게 바꿨다" (after-image) "원래 이랬다" (before-image)
ACID 역할 Durability Atomicity + Isolation
장애 복구 커밋된 변경을 재실행 (Roll Forward) 미완료 변경을 취소 (Roll Back)
쓰기 방식 Sequential (Append) 페이지 내 제자리 쓰기
정리 시점 Checkpoint 완료 후 재사용 모든 참조 트랜잭션 종료 후 Purge
실무 주의 크기 부족 → Checkpoint 폭주 장시간 Tx → Purge 지연, 디스크 폭증
"DB는 메모리에서 일하고, 로그로 보증한다."
Redo Log가 커밋의 내구성을 보장하고, Undo Log가 롤백과 동시성을 보장한다. 이 두 로그의 협력이 트랜잭션의 ACID를 완성한다.

KEY TAKEAWAY

COMMIT은 데이터 파일을 바꾸는 게 아니라 Redo Log를 디스크에 쓰는 것이다.
ROLLBACK은 Redo Log를 되감는 게 아니라 Undo Log에서 원래 값을 복원하는 것이다.
이 구분을 이해하면 DB 내부 동작의 절반이 보인다.
