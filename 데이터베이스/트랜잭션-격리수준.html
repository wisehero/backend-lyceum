TRANSACTION
ISOLATION

동시에 실행되는 트랜잭션들이 서로를 얼마나 볼 수 있는가.
정합성과 동시성의 줄다리기를 어디서 끊을지 정하는 규칙이다.

01 ANOMALIES

격리가 부족하면 생기는 3가지 문제

트랜잭션이 동시에 실행될 때, 아무런 규칙 없이 서로의 데이터를 자유롭게 읽고 쓰면 다음 세 가지 이상 현상이 발생한다.

Dirty Read — 커밋 안 된 데이터를 읽는 문제

TX1
TX2
UPDATE  balance = 500
—
—
SELECT  balance → 500
COMMIT 전인데 읽어버림
ROLLBACK
—
—
TX2가 읽은 500은 존재하지 않는 데이터

Non-Repeatable Read — 같은 쿼리인데 결과가 다른 문제

TX1
TX2
SELECT  balance → 1000
—
—
UPDATE  balance = 500
—
COMMIT
SELECT  balance → 500
같은 쿼리인데 결과가 바뀜
—

Dirty Read와 달리 TX2가 정상적으로 COMMIT한 유효한 데이터다. 하지만 하나의 트랜잭션 안에서 일관성이 깨졌다.

Phantom Read — 없던 행이 나타나는 문제

TX1
TX2
SELECT  WHERE age > 20 → 3건
—
—
INSERT  (age=25) 새 행 추가
—
COMMIT
SELECT  WHERE age > 20 → 4건
없던 행이 나타남 (Phantom)
—

Non-Repeatable Read는 기존 Row의 값이 바뀌는 것이고, Phantom Read는 행 자체가 추가/삭제되어 결과 집합이 변한다.

이상 현상무엇이 변하나원인위험도
Dirty Read UNCOMMITTED 값을 읽음 다른 TX가 COMMIT 전에 읽음 높음
Non-Repeatable Read 같은 Row를 다시 읽었더니 값이 다름 다른 TX가 UPDATE + COMMIT 중간
Phantom Read 같은 조건인데 행 수가 다름 다른 TX가 INSERT/DELETE + COMMIT 낮음
02 ISOLATION LEVELS

네 가지 격리 수준

SQL 표준은 세 가지 이상 현상을 어디까지 허용하느냐에 따라 네 가지 격리 수준을 정의한다.

LEVEL 0

READ UNCOMMITTED

다른 TX가 COMMIT하지 않은 변경도 보인다. 격리가 거의 없는 수준.

실무에서 거의 사용하지 않음
LEVEL 1

READ COMMITTED

COMMIT된 데이터만 보인다. 하지만 같은 TX 안에서 결과가 달라질 수 있다.

Oracle, PostgreSQL 기본
LEVEL 2

REPEATABLE READ

TX 시작 시점의 Snapshot 기준으로 읽는다. 같은 쿼리는 항상 같은 결과.

MySQL (InnoDB) 기본
LEVEL 3

SERIALIZABLE

트랜잭션이 하나씩 순서대로 실행되는 것처럼 동작한다. 완벽한 격리.

금융 정산 등 특수 상황에서만

격리 수준별 이상 현상 허용 여부

Isolation LevelDirty ReadNon-Repeatable ReadPhantom Read동시성
READ UNCOMMITTED 발생 발생 발생 매우 높음
READ COMMITTED 방지 발생 발생 높음
REPEATABLE READ 방지 방지 발생 (InnoDB: 대부분 방지) 보통
SERIALIZABLE 방지 방지 방지 낮음
핵심: 격리 수준이 올라갈수록 데이터 정합성은 높아지지만 동시 처리 성능은 떨어진다. 정합성이냐 동시성이냐, 이 줄다리기를 어디서 끊을지 정하는 게 Isolation Level이다.
03 RC vs RR

READ COMMITTED vs REPEATABLE READ

실무에서 가장 많이 마주치는 두 격리 수준의 핵심 차이는 Snapshot 생성 시점이다.

READ COMMITTED

매 SELECT마다 새로운 Snapshot을 생성한다. 항상 최신 COMMIT 기준으로 읽는다.

SELECT 1
S1
SELECT 2
S2
SELECT 3
S3

매번 새 Snapshot → 결과가 달라질 수 있음

REPEATABLE READ

첫 번째 읽기 시 단 한 번 Snapshot을 생성하고, TX 끝까지 재사용한다.

SELECT 1
S1
SELECT 2
—
SELECT 3
—

처음 한 번만 → 항상 같은 결과

비교READ COMMITTEDREPEATABLE READ
Snapshot 생성 시점 매 쿼리(SELECT)마다 첫 번째 읽기 시 단 한 번
같은 TX 안에서 같은 쿼리 결과가 다를 수 있음 항상 같은 결과
Undo Log 참조 최신 COMMIT 기준 TX 시작 시점 기준
기본 DB Oracle, PostgreSQL MySQL (InnoDB)

InnoDB의 Phantom Read 방지: Next-Key Lock

SQL 표준에서는 REPEATABLE READ에서 Phantom Read가 발생할 수 있지만, InnoDB는 Next-Key Lock(Record Lock + Gap Lock)으로 대부분 방지한다.

주의 — 읽기 방식에 따라 다르다:
일반 SELECT: MVCC Snapshot으로 Phantom 방지 (스냅샷 이후 INSERT된 행이 안 보임)
SELECT ... FOR UPDATE: Next-Key Lock으로 Phantom 방지 (INSERT 자체를 차단)
두 방식을 혼용하면 서로 다른 기준으로 읽기 때문에 결과가 달라질 수 있다.
04 MVCC

MVCC — Isolation의 구현 메커니즘

격리 수준은 MVCC(Multi-Version Concurrency Control)로 구현된다. 데이터를 변경할 때 이전 버전을 Undo Log에 보관하고, 읽는 TX은 자신에게 맞는 버전을 골라 읽는다.

MVCC 없는 세계

쓰기 트랜잭션이 Lock을 잡으면, 읽기 트랜잭션도 Lock이 풀릴 때까지 대기해야 한다. 읽기와 쓰기가 서로 블로킹된다.

MVCC 있는 세계

읽는 쪽은 Undo Log에서 이전 버전을 읽고, 쓰는 쪽은 최신 데이터를 수정한다. 읽기와 쓰기가 서로 안 기다린다.

Read View (스냅샷): 트랜잭션이 "어느 시점의 데이터를 볼 것인가"를 결정하는 기준. 현재 활성 트랜잭션 목록을 기록하여, 어떤 변경이 보이고 어떤 변경이 안 보이는지 판단한다.

Undo Log와의 연결: MVCC는 Undo Log의 버전 체인을 따라가며 자신의 Read View에 맞는 과거 버전을 찾는다. REPEATABLE READ는 TX 전체 기간 동안 Read View를 유지하므로, Undo Log도 그만큼 오래 보관되어야 한다.
05 LOCK & CONCURRENCY

동시성 문제는 Lock으로 해결한다

Isolation Level은 읽기 일관성 이야기다. 동시 쓰기 충돌은 Lock이나 원자적 연산으로 따로 해결해야 한다.

Lost Update: 동시 재고 차감 시나리오

재고 1개 남은 상품에 두 명이 동시에 주문. REPEATABLE READ여도 이 문제는 발생한다.

TX1
TX2
SELECT  stock → 1 (재고 있음)
—
—
SELECT  stock → 1 (재고 있음)
UPDATE  stock = 0
—
COMMIT
—
—
UPDATE  stock = 0
COMMIT
둘 다 성공 → 실제로는 2번 차감됨
원인: MVCC Snapshot은 읽기에만 적용된다. 쓰기(UPDATE)는 항상 최신 데이터에 적용되므로 두 TX가 동시에 stock=1에서 1을 빼면 0이 두 번 쓰여진다.

비관적 Lock (Pessimistic)

@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p
        WHERE p.id = :id")
Product findByIdForUpdate(
    @Param("id") Long id);

SELECT ... FOR UPDATE로 행을 먼저 잠근다. 다른 TX는 Lock이 풀릴 때까지 대기.

낙관적 Lock (Optimistic)

@Entity
public class Product {
    @Version
    private Long version;
}
// UPDATE product
//   SET stock=9, version=2
//   WHERE id=1 AND version=1
// → version 불일치 시 예외

@Version으로 커밋 시점에 충돌을 감지한다. 충돌 시 재시도 로직 필요.

방식동작장점단점적합한 상황
Pessimistic Lock SELECT 시점에 Row Lock 획득 확실한 순서 보장 Lock 대기 성능 저하 충돌이 자주 발생
Optimistic Lock COMMIT 시점에 Version 비교 Lock 없이 높은 동시성 재시도 로직 필요 충돌이 드묾
원자적 UPDATE DB에서 조건부 차감 가장 효율적 복잡한 로직 어려움 단순 차감 연산 
// 방법 3: 원자적 UPDATE — 가장 효율적
@Modifying
@Query("UPDATE Product p SET p.stock = p.stock - 1
        WHERE p.id = :id AND p.stock > 0")
int decreaseStock(@Param("id") Long id);
// 반환값 0 → 재고 부족으로 처리
"SERIALIZABLE은 전부 다 Lock이고, SELECT FOR UPDATE는 필요한 Row만 Lock이니까 후자가 훨씬 효율적이다."
Isolation Level을 올리는 것보다 필요한 곳에만 Lock을 거는 게 낫다.
06 PRACTICE

실무 가이드

대부분의 경우 DB 기본값을 그대로 쓰는 게 맞다. Isolation Level을 올려야 하는 경우는 매우 제한적이다.

Spring/JPA에서 격리 수준 설정

// 기본값: DB의 기본 Isolation Level을 따름
@Transactional
public void defaultIsolation() { ... }

// 명시적으로 Isolation Level 지정
@Transactional(isolation = Isolation.READ_COMMITTED)
public void readCommitted() { ... }

@Transactional(isolation = Isolation.REPEATABLE_READ)
public void repeatableRead() { ... }
주의: @Transactional(isolation = ...)은 해당 TX에만 적용되고, TX 종료 후 이전 격리 수준으로 복원된다. 또한 기존 TX에 참여하는 경우(PROPAGATION_REQUIRED로 합류)에는 격리 수준 지정이 적용되지 않는다.

Isolation Level을 올리는 것보다 나은 대안

상황고려할 수 있는 수준더 나은 대안
재고 차감 시 동시 요청 SERIALIZABLE 비관적 Lock (SELECT FOR UPDATE)
포인트 잔액 조회 후 차감 SERIALIZABLE 낙관적 Lock (Version 컬럼)
집계 보고서 일관성 REPEATABLE READ @Transactional(readOnly=true)

DB별 기본 격리 수준

DB기본 Isolation LevelMVCC특이사항
MySQL (InnoDB) REPEATABLE READ 지원 Next-Key Lock으로 Phantom Read 대부분 방지
Oracle READ COMMITTED 지원 REPEATABLE READ 미지원. RC / SERIALIZABLE / READ ONLY만 제공
PostgreSQL READ COMMITTED 지원 RR에서 write-write 충돌 감지 시 오류 반환
SQL Server READ COMMITTED 선택적 Snapshot Isolation이라는 별도 수준 제공

DB별 기본값 차이

프로젝트에서 쓰는 DB가 뭐냐에 따라 기본 동작이 달라진다. MySQL에서 개발하다가 Oracle로 옮기면 격리 동작이 달라질 수 있다.

PostgreSQL 주의

PostgreSQL의 RR은 write-write 충돌 시 오류를 반환한다. MySQL InnoDB는 이런 감지가 없으므로, 애플리케이션에서 Lock이나 원자적 UPDATE로 직접 해결해야 한다.

KEY TAKEAWAY

Isolation Level은 읽기 일관성을 결정하는 규칙이다.
쓰기 충돌은 Lock이나 원자적 연산으로 따로 해결해야 한다.
대부분의 경우 DB 기본값을 유지하고, 필요한 곳에만 Lock을 거는 것이 정답이다.
