01 PROBLEM

Dual Write Problem

DB 커밋과 Kafka 전송을 원자적으로 묶을 수 없는 근본적 한계

Dual Write란? DB 커밋과 Kafka 전송은 서로 다른 시스템에 대한 쓰기이므로 하나의 로컬 트랜잭션으로 원자적(atomic)으로 묶을 수 없다. 두 시스템 모두 성공하거나 모두 실패하는 것을 보장할 방법이 없기 때문에 데이터 불일치가 발생한다. 
@Transactional
public void createOrder(OrderRequest request) {
    Order order = orderRepository.save(new Order(request));       // 1) DB 쓰기
    kafkaTemplate.send("order-events", new OrderCreatedEvent(order)); // 2) Kafka 쓰기 — 위험!
    // DB 커밋과 Kafka 전송은 별개의 시스템 → 하나만 실패할 수 있다
}

DB 성공 + Kafka 실패

주문은 저장되었지만 이벤트가 발행되지 않아 다운스트림 서비스(재고, 알림, 결제)가 주문을 인지하지 못한다. 네트워크 장애, Kafka 브로커 다운, 타임아웃 등이 원인이다.

Kafka 성공 + DB 롤백

Kafka에는 이벤트가 전송되었지만 이후 DB 커밋이 실패하여 롤백된다. 존재하지 않는 주문에 대한 이벤트가 전파되어 데이터 불일치가 발생한다.

참고: Kafka KIP-939는 Kafka와 외부 시스템 간 2PC(Two-Phase Commit)를 시도하고 있지만 아직 릴리스되지 않았다. 현시점에서 Dual Write 문제를 해결하는 가장 실용적인 방법이 Outbox 패턴이다.
Outbox 패턴의 핵심 아이디어: 메시지를 외부 브로커에 직접 보내지 말고, 같은 DB 트랜잭션 안에서 Outbox 테이블에 저장한 뒤, 별도 프로세스가 이를 읽어 Kafka로 발행한다. DB의 ACID 트랜잭션이 원자성을 보장하므로, 비즈니스 데이터와 이벤트가 반드시 함께 저장되거나 함께 롤백된다.

03 TABLE DESIGN

Outbox 테이블 설계

이벤트 발행에 필요한 모든 정보를 담는 테이블 구조

CREATE TABLE outbox_event (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    aggregate_type  VARCHAR(255) NOT NULL,   -- e.g., 'Order'
    aggregate_id    VARCHAR(255) NOT NULL,   -- e.g., 주문 ID
    event_type      VARCHAR(255) NOT NULL,   -- e.g., 'OrderCreated'
    payload         TEXT NOT NULL,            -- JSON 직렬화된 이벤트 본문
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    published       BOOLEAN NOT NULL DEFAULT FALSE
);

CREATE INDEX idx_outbox_unpublished
    ON outbox_event (published, created_at);

aggregate_type + aggregate_id

이 이벤트가 어떤 도메인 엔티티에서 발생했는지 식별한다. aggregate_type은 Kafka 토픽 라우팅에, aggregate_id는 Kafka 파티션 키로 활용하여 같은 Aggregate의 이벤트 순서를 보장한다.

published + 복합 인덱스

Polling Publisher에서 미발행 이벤트를 효율적으로 조회하기 위한 설계이다. (published, created_at) 복합 인덱스로 WHERE published = FALSE ORDER BY created_at 쿼리의 성능을 최적화한다.

event_type

이벤트의 종류를 식별한다. Consumer가 자신이 관심 있는 이벤트만 필터링하여 처리할 수 있게 한다. OrderCreated, OrderCancelled 등 도메인 이벤트명을 저장한다.

payload (JSON)

이벤트 본문을 JSON 문자열로 직렬화하여 저장한다. Consumer가 필요로 하는 모든 정보를 포함해야 하며, 스키마 변경 시 하위 호환성을 유지하는 것이 중요하다.

04 IMPLEMENTATION

비즈니스 로직 구현

비즈니스 데이터와 Outbox 이벤트를 하나의 트랜잭션으로 저장한다

@Service
@RequiredArgsConstructor
public class OrderService {
    private final OrderRepository orderRepository;
    private final OutboxEventRepository outboxEventRepository;
    private final ObjectMapper objectMapper;

    @Transactional
    public Order createOrder(OrderRequest request) {
        // 1) 비즈니스 엔티티 저장
        Order order = Order.from(request);
        orderRepository.save(order);

        // 2) Outbox 이벤트 저장 (같은 트랜잭션)
        String payload = objectMapper.writeValueAsString(
            new OrderCreatedEvent(order)
        );
        OutboxEvent outboxEvent = OutboxEvent.create(
            "Order",
            order.getId().toString(),
            "OrderCreated",
            payload
        );
        outboxEventRepository.save(outboxEvent);

        return order;
    }
}
핵심: orderRepository.save()와 outboxEventRepository.save()가 같은 @Transactional 안에서 실행되므로, 둘 다 성공하거나 둘 다 실패한다. DB의 ACID 보장이 곧 이벤트 저장의 신뢰성이 된다.
주의: Outbox 이벤트 저장은 반드시 비즈니스 로직과 동일한 트랜잭션 경계 안에 있어야 한다. 별도의 @Async나 이벤트 리스너로 분리하면 원자성이 깨진다.
05 MESSAGE RELAY

Polling Publisher 구현

주기적으로 Outbox 테이블을 조회하여 미발행 이벤트를 Kafka로 전송한다

@Component
@RequiredArgsConstructor
@Slf4j
public class OutboxMessageRelay {
    private final OutboxEventRepository outboxEventRepository;
    private final KafkaTemplate<String, String> kafkaTemplate;

    @Scheduled(fixedDelay = 5000)
    @SchedulerLock(name = "outbox-relay",
        lockAtLeastFor = "PT4S", lockAtMostFor = "PT30S")
    @Transactional
    public void publishOutboxEvents() {
        List<OutboxEvent> events =
            outboxEventRepository.findTop100ByPublishedFalseOrderByCreatedAtAsc();

        for (OutboxEvent event : events) {
            try {
                String topic = resolveTopicName(event.getAggregateType());
                String key = event.getAggregateId();
                kafkaTemplate.send(topic, key, event.getPayload());
                event.markPublished();
            } catch (Exception e) {
                log.error("Error processing outbox event id={}",
                    event.getId(), e);
            }
        }
    }
}

lockAtLeastFor: PT4S

한 인스턴스가 작업을 빨리 끝내더라도 최소 4초간 락을 유지하여 다른 인스턴스의 중복 실행을 방지한다. fixedDelay(5s)보다 짧게 설정하여 간격 내 단 한 번만 실행되도록 한다.

lockAtMostFor: PT30S

인스턴스가 비정상 종료(OOM, kill -9)되면 락이 자동으로 해제되지 않는다. 최대 30초 후 락을 강제 해제하여 다른 인스턴스가 작업을 이어받을 수 있게 한다.

aggregateId를 Kafka key로 사용하라: 같은 Aggregate의 이벤트가 동일 파티션에 배치되어 순서가 보장된다. Kafka는 파티션 내에서만 순서를 보장하므로, 순서가 중요한 이벤트는 반드시 같은 key를 사용해야 한다.
06 OPERATIONS

Consumer 멱등성

At-Least-Once 보장 구조이므로 Consumer 측의 멱등성 처리가 필수이다

왜 멱등성이 필요한가? Relay가 Kafka 전송에 성공했지만 published 플래그를 업데이트하기 전에 장애가 발생하면, 재시작 후 같은 이벤트를 다시 전송한다. 따라서 Consumer는 동일 이벤트를 여러 번 받아도 결과가 동일하도록 멱등성을 보장해야 한다. 
@KafkaListener(topics = "order-events", groupId = "inventory-service")
@Transactional
public void handleOrderCreated(ConsumerRecord<String, String> record) {
    OrderCreatedEvent event = deserialize(record.value());

    // 중복 체크: 이미 처리된 이벤트인지 확인
    if (processedEventRepository.existsByEventId(event.getEventId())) {
        log.info("Duplicate event detected, skipping. eventId={}",
            event.getEventId());
        return;
    }

    // 비즈니스 로직 실행
    inventoryService.decreaseStock(event.getProductId(), event.getQuantity());

    // 처리 완료 기록 (같은 트랜잭션)
    processedEventRepository.save(new ProcessedEvent(event.getEventId()));
}

Kafka Producer 설정

acks=all: 모든 ISR 복제본이 메시지를 기록한 후 응답
enable.idempotence=true: 네트워크 재시도로 인한 메시지 중복 방지
retries=Integer.MAX_VALUE: 일시적 장애 시 무한 재시도

Outbox 테이블 정리

published=true인 오래된 레코드를 주기적 배치로 삭제해야 한다. 정리하지 않으면 테이블이 비대해져 인덱스 성능이 저하되고 디스크가 증가한다.

07 ANTI-PATTERNS

안티 패턴

Outbox 없이 이벤트를 발행하려는 시도들의 한계

@TransactionalEventListener(AFTER_COMMIT)

AFTER_COMMIT 시점에서는 원래 트랜잭션이 이미 커밋되었지만, thread-local에 트랜잭션 리소스가 아직 남아있다. 기본 전파(REQUIRED)로 DB 쓰기를 시도하면, 이미 커밋된 트랜잭션에 참여하여 쓰기가 조용히 유실된다. REQUIRES_NEW를 명시해도 비즈니스 로직과의 원자성이 깨지므로 안티패턴이다. 이벤트 리스너에서 Kafka 전송이 실패하면 원래 트랜잭션은 이미 커밋된 상태이므로 재시도 메커니즘이 없다.

Spring @Async + ApplicationEventPublisher

메모리 기반 이벤트 처리로, 애플리케이션이 재시작되면 미처리 이벤트가 모두 유실된다. 프로세스 경계를 넘지 못하며, 비동기 스레드 풀 고갈 시 이벤트가 누락될 수 있다.

Spring Modulith Event Publication Registry: Spring Modulith는 이벤트 발행 기록을 DB에 영속화하여 미처리 이벤트를 재시도할 수 있는 메커니즘을 제공한다. 내부 모듈 간 이벤트에는 유용하지만, 외부 브로커(Kafka)로의 발행 보장이 필요한 경우에는 Outbox 패턴이 더 명시적이고 제어 가능하다.
Fat Event vs Thin Event: Fat Event는 이벤트에 전체 데이터를 포함하여 Consumer가 독립적으로 처리할 수 있지만 payload 크기가 커진다. Thin Event는 ID만 포함하여 가볍지만 Consumer가 원본 데이터를 다시 조회해야 하며, 조회 시점의 데이터가 이벤트 발생 시점과 달라질 수 있다(Temporal Coupling). 일반적으로 Consumer의 독립성이 중요한 MSA 환경에서는 Fat Event가 선호된다.

CDC 고유 위험: binlog/WAL 보존

MySQL binlog_expire_logs_seconds 기본값은 30일이다. CDC가 이 기간 이상 중단되면 binlog가 만료되어 이벤트가 유실된다. PostgreSQL의 WAL replication slot은 CDC가 중단되면 WAL이 삭제되지 않아 디스크가 고갈될 수 있다.

Debezium Outbox SMT

io.debezium.transforms.outbox.EventRouter는 Outbox 테이블의 레코드를 자동으로 파싱하여 aggregate_type 기반 토픽으로 라우팅하고, aggregate_id를 Kafka 키로 설정하는 전용 SMT이다.

현실적 선택: 소규모 서비스라면 Polling으로 시작하고, 처리량 증가에 따라 CDC로 전환하는 것이 현실적이다. Polling의 단순함은 초기 운영 부담을 크게 줄여준다.

모놀리스에서의 Outbox

Kafka 없이도 Outbox 패턴의 핵심 원리는 동일하게 적용된다

왜 모놀리스에도 Outbox가 필요한가? 모놀리스에서도 주문 생성 후 이메일 발송, Slack 알림, PDF 생성 같은 부수 효과(side effect)가 있다. 이들을 동기적으로 실행하면 외부 서비스 장애가 주문 생성을 실패시키고, 비동기로 실행하면 애플리케이션 재시작 시 유실된다. Outbox 테이블에 작업을 기록하면 부수 효과를 신뢰성 있게 실행할 수 있다.

CREATE TABLE outbox_task (
    id            BIGINT AUTO_INCREMENT PRIMARY KEY,
    task_type     VARCHAR(100) NOT NULL,    -- EMAIL, SLACK, PDF_GENERATE
    reference_id  VARCHAR(255) NOT NULL,   -- 관련 엔티티 ID
    payload       TEXT NOT NULL,
    status        VARCHAR(20) DEFAULT 'PENDING', -- PENDING, DONE, FAILED
    retry_count   INT DEFAULT 0,
    max_retries   INT DEFAULT 3,
    next_retry_at TIMESTAMP,
    last_error    TEXT,
    created_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
// Exponential Backoff 재시도 로직
public void markFailed(String errorMessage) {
    this.retryCount++;
    this.lastError = errorMessage;
    if (this.retryCount >= this.maxRetries) {
        this.status = TaskStatus.FAILED;
    } else {
        this.status = TaskStatus.PENDING;
        long backoffMinutes = (long) Math.pow(2, this.retryCount);
        this.nextRetryAt = LocalDateTime.now().plusMinutes(backoffMinutes);
    }
}
10 MONOLITH CODE

모놀리스 구현 코드

비즈니스 로직에서 다중 작업을 등록하고 Strategy 패턴으로 처리한다

@Transactional
public Order createOrder(OrderRequest request) {
    Order order = Order.from(request);
    orderRepository.save(order);

    String orderId = order.getId().toString();
    String payload = toJson(new OrderCreatedPayload(order));

    // 하나의 트랜잭션에서 여러 작업을 동시 등록
    outboxTaskRepository.save(
        OutboxTask.create("SEND_ORDER_CONFIRMATION_EMAIL", orderId, payload, 3));
    outboxTaskRepository.save(
        OutboxTask.create("NOTIFY_SLACK_CHANNEL", orderId, payload, 2));
    outboxTaskRepository.save(
        OutboxTask.create("GENERATE_INVOICE_PDF", orderId, payload, 5));

    return order;
}
// Strategy Pattern으로 작업 타입별 핸들러 분리
public interface OutboxTaskHandler {
    String taskType();
    void handle(OutboxTask task);
}

@Component
@RequiredArgsConstructor
public class OutboxProcessor {
    private final Map<String, OutboxTaskHandler> handlers;
    private final OutboxTaskRepository taskRepository;

    @Scheduled(fixedDelay = 3000)
    @Transactional
    public void process() {
        List<OutboxTask> tasks = taskRepository
            .findPendingTasks(LocalDateTime.now());

        for (OutboxTask task : tasks) {
            try {
                handlers.get(task.getTaskType()).handle(task);
                task.markDone();
            } catch (Exception e) {
                task.markFailed(e.getMessage());
            }
        }
    }
}

FAILED 모니터링

max_retries를 초과한 FAILED 작업은 Dead Letter Queue 역할을 한다. 알림을 설정하고 원인을 분석하여 수동 재처리하라.

수동 재시도 API

관리자가 FAILED 상태의 작업을 수동으로 PENDING으로 변경하여 재시도할 수 있는 Admin API를 제공하라.

Modular Monolith 확장

모듈 간 Outbox 패턴을 적용하면, 나중에 MSA로 전환할 때 Outbox 테이블을 Kafka 발행으로 자연스럽게 교체할 수 있다.

초고처리량 환경에서 모든 이벤트가 Outbox를 거치면 RDB 쓰기 성능이 상한이 된다. 이 경우 Listen-to-Yourself 패턴(자신이 발행한 이벤트를 스스로 구독)이나 Event Sourcing으로의 전환을 검토하라.

순서 보장 한계

aggregate_id를 Kafka key로 사용하면 파티션 내 순서는 보장되지만, Relay의 재시도나 다중 Relay 인스턴스에서 순서가 역전될 수 있다. 엄격한 순서가 필요하면 단일 Relay 또는 직렬 처리가 필요하다.

스키마 결합

이벤트 payload 스키마를 변경하면 Consumer가 깨질 수 있다. 새 필드는 반드시 Optional로 추가하고, 기존 필드의 이름이나 타입을 변경하지 마라. 하위 호환성을 반드시 유지해야 한다.

디버깅 복잡성

이벤트의 흐름을 추적하기 어렵다. 반드시 correlation ID를 payload에 포함하여 전파하라. Producer에서 생성한 ID가 Consumer까지 이어져야 전체 흐름을 추적할 수 있다.

Saga 패턴 혼동

Outbox는 이벤트 발행의 신뢰성을 보장하는 패턴이고, Saga는 다중 서비스 간 트랜잭션의 최종 일관성을 보장하는 패턴이다. 목적이 다른 별개 패턴이며, Saga의 각 단계에서 Outbox를 사용할 수 있다.

테이블 관리 부담

Housekeeping 없이 방치하면 Outbox 테이블이 무한히 커져 인덱스 성능이 저하되고 디스크 사용량이 증가한다. 발행 완료된 레코드를 주기적으로 삭제하는 배치 작업이 반드시 필요하다.

12 SUMMARY

핵심 정리

Dual Write 해결

DB 트랜잭션의 원자성을 활용하여 비즈니스 데이터와 이벤트를 함께 저장한다. 외부 브로커에 직접 쓰지 않으므로 원자성이 보장된다.

At-Least-Once

이벤트가 최소 한 번은 전달되므로 Consumer 측의 멱등성 처리가 반드시 필요하다. processed_events 테이블로 중복을 감지하라.

도입 판단

Dual Write의 심각도에 비례하는 복잡성을 감수해야 한다. 모든 상황에 필요한 것은 아니며, 비용 대비 효과를 따져야 한다.

도입 전 질문: "우리 시스템에서 Dual Write 문제가 실제로 얼마나 심각한가?"를 먼저 판단하라. 이벤트 유실이 비즈니스에 치명적인 경우(결제, 재고)에만 Outbox 패턴의 복잡성이 정당화된다.
"Outbox 패턴은 분산 시스템의 데이터 일관성 문제를 해결하는 실용적인 패턴이지만, 공짜 점심은 아니다."

TRANSACTION OUTBOX PATTERN

핵심은 비즈니스 데이터 변경과 이벤트 기록의 원자적 저장이다.
단일 트랜잭션의 ACID 보장을 이벤트 발행의 신뢰성으로 전환하는 것이 이 패턴의 본질이다.
