DATA
CONSISTENCY

정합성을 강하게 보장할수록 성능과 가용성을 희생해야 한다

01 DEFINITION

정합성이란 무엇인가

데이터가 서로 모순 없이 일관된 상태를 유지하고 있는 것

정합성(Consistency) — 시스템이 정해놓은 규칙(불변 조건)이 항상 지켜지고 있는 상태.
예: 은행 송금에서 A 계좌에서 빠진 금액과 B 계좌에 들어온 금액은 반드시 같아야 한다. 이 규칙이 깨지면 정합성이 깨진 것이다.

데이터 정합성

데이터 자체가 비즈니스 규칙에 맞는가?

위반 예시: 주문 상태가 "배송완료"인데 배송 기록이 없음.
하나의 DB 안에서도 발생할 수 있다.

읽기 정합성

누군가 데이터를 읽을 때 최신 상태를 정확히 보는가?

위반 예시: 관리자가 가격을 수정했는데, 다른 사용자가 이전 값을 본다.
분산 환경에서 본격적으로 깊어진다.

02 ACID

ACID의 C를 제대로 이해하기

트랜잭션 전에 유효했다면, 트랜잭션 후에도 유효해야 한다

ACID의 C (Consistency)

"유효한 상태"란 DB에 정의된 모든 제약 조건이 만족되는 상태를 말한다.
예: 잔고 >= 0 제약이 걸린 계좌에서 5만 원 잔고로 10만 원 출금을 시도하면, DB가 이를 거부하여 유효한 상태를 유지한다.

ACID의 C는 DB 혼자 지키는 게 아니다. DB는 NOT NULL, UNIQUE, FK, CHECK만 지킬 수 있고, "잔고 차감 시 상대 잔고 증가" 같은 비즈니스 규칙은 애플리케이션이 보장해야 한다.

1. 트랜잭션 미묶음

송금 시 출금과 입금을 하나의 TX로 묶지 않으면 중간에 장애가 발생했을 때 출금만 되고 입금은 안 될 수 있다.

2. 동시성 문제

두 사용자가 동시에 재고를 차감할 때 격리 수준이 부적절하면 마이너스 재고가 발생할 수 있다.

3. 애플리케이션 버그

주문 취소 시 재고 원복은 했는데 포인트 환불을 누락하는 경우. DB 제약으로는 잡을 수 없다.

03 CAP

분산 환경에서의 정합성

데이터가 여러 노드에 흩어지면 "어느 노드가 진짜인가?"라는 질문이 생긴다

왜 어려운가?

네트워크는 본질적으로 불안정하다 — 지연(Latency), 패킷 유실, 노드 장애가 언제든 발생한다. 단일 DB에서 락과 트랜잭션으로 해결하던 것이, 노드 간 통신이 끼면 근본적으로 다른 문제가 된다.

Consistency

모든 노드가 같은 시점에 같은 데이터를 본다.

Availability

모든 요청이 에러 아닌 응답을 받는다.

Partition Tolerance

네트워크 단절되어도 시스템이 동작한다.

P는 포기할 수 없다. 네트워크 파티션은 반드시 발생한다. 따라서 실질적 선택은 CP vs AP이다.

CP 시스템

"잘못된 데이터를 줄 바에는 에러를 돌려주겠다"

대표: 은행 계좌, ZooKeeper, etcd

AP 시스템

"다소 오래된 데이터라도 응답 안 하는 것보다 낫다"

대표: SNS 타임라인, Cassandra, DynamoDB

CAP 선택의 실제 — CAP 선택은 네트워크 파티션 발생 시에만 적용된다. 정상 상태에서는 C와 A 모두 가능하다. 기능별로 다른 선택도 가능하다 — 결제는 CP, 추천은 AP.
04 SPECTRUM

정합성의 스펙트럼

정합성은 있다/없다가 아니라 강한 것에서 느슨한 것까지 스펙트럼이다

Strong Consistency

쓰기 완료 후 모든 읽기가 최신 값을 반환한다. 모든 노드 확인까지 대기하므로 느리고, 가용성이 저하된다.

대표: Raft, Paxos 합의 알고리즘 기반 시스템

Causal Consistency

인과 관계 있는 작업만 순서를 보장한다. 질문 → 답변 순서는 보장되지만, 무관한 글은 노드마다 달라도 OK.

인과 관계 추적에 벡터 클럭 등 사용

Read-Your-Writes

내가 쓴 건 내가 읽을 때 최신으로 보인다. 다른 사용자에게는 즉시 반영이 보장되지 않는다.

Sticky Session, 쓰기 후 Primary 읽기 등으로 구현

Eventual Consistency

지금은 다를 수 있지만 시간이 지나면 결국 수렴한다. "언제"는 보장하지 않는다.

대표: DynamoDB, Cassandra (ConsistentRead / QUORUM으로 강한 일관성도 지원)
"정합성은 공짜가 아니다. 강하게 보장할수록 성능과 가용성을 희생해야 한다. 좋은 아키텍트는 '이 데이터에는 어느 수준의 정합성이 적절한가?'를 판단할 수 있는 사람이다."
05 SAGA

2PC와 Saga 패턴

MSA에서 여러 서비스의 DB를 하나의 트랜잭션으로 묶는 것은 불가능하다

2PC (Two-Phase Commit)

Prepare 단계 — 코디네이터가 모든 참여자에게 "커밋 준비 됐어?" 확인
Commit 단계 — 모두 OK면 "커밋해" 지시

블로킹

모든 참여자 응답까지 락 유지. 하나라도 느리면 전체가 블로킹.

코디네이터 장애

코디네이터가 죽으면 참여 노드가 in-doubt 상태에 빠진다.

강한 결합

Saga 패턴

큰 분산 트랜잭션을 로컬 TX + 보상 TX로 분해한다. 각 단계는 자기 DB에서 로컬 트랜잭션으로 처리하고, 실패 시 역순으로 보상 트랜잭션을 실행한다.

Choreography

각 서비스가 이벤트를 발행/구독. 결합도가 낮지만 전체 흐름 파악이 어렵다.

Orchestration

06 EVENTS

Outbox와 Event Sourcing

이벤트 발행의 신뢰성과 변경 이력의 완전한 추적

Transactional Outbox 패턴

문제: DB 저장 성공 후 이벤트 발행 전에 서비스가 죽으면 정합성이 깨진다.
해결: 같은 TX에서 비즈니스 데이터와 Outbox 테이블에 이벤트를 함께 저장한다.

Polling Publisher

주기적으로 Outbox 테이블을 조회하여 발행. 단순하지만 지연이 있다.

CDC (Debezium)

DB 변경 로그를 실시간 캡처하여 발행. 지연이 적지만 인프라가 복잡하다.

중복 발행 가능성이 존재한다. Relay가 브로커에 전달한 후 Outbox에서 삭제하기 전에 장애가 나면 같은 이벤트가 다시 발행된다. Consumer 측 멱등성(Idempotency) 보장은 필수이다.

Event Sourcing

현재 상태 대신 모든 변경 이벤트를 순서대로 저장한다. 현재 상태는 이벤트 재생(Replay)으로 계산한다.

완전한 이력 (Audit Trail)

"언제, 왜 이 상태가 됐는지" 추적 가능. 규제 준수와 디버깅에 강력하다.

불변성 (Immutable)

이벤트는 Append-only. 수정/삭제 없으므로 데이터 충돌을 원천 차단한다.

07 BIG PICTURE

CQRS와 패턴들의 관계

쓰기와 읽기를 분리하고, 모든 패턴이 어떻게 연결되는지

Command (쓰기)

정규화된 모델로 비즈니스 규칙을 검증한다. 데이터 무결성에 집중.

Query (읽기)

비정규화된 뷰로 빠른 조회를 제공한다. 읽기 성능에 최적화.

쓰기 → 읽기 모델은 비동기로 업데이트된다. 쓰기 완료 후에도 읽기 모델에 바로 반영되지 않는 일시적 불일치가 존재한다. CQRS도 Eventual Consistency를 수용하는 패턴이다.
1 분산 트랜잭션이 불가능하다 → Saga로 로컬 TX를 조합
2 Saga 각 단계 이벤트를 안전하게 발행해야 한다 → Outbox 패턴
3 모든 변경을 이벤트로 기록하고 싶다 → Event Sourcing
4 쓰기와 읽기의 요구사항이 다르다 → CQRS
모든 패턴의 공통 원칙 — 강한 정합성 대신 최종 정합성을 수용하되, 그 과정을 안전하고 추적 가능하게 관리한다. 소비자 측의 멱등성이 기본 전제.
08 SUMMARY

핵심 정리

1단계: 공동 책임

정합성은 DB와 애플리케이션의 공동 책임이다. DB는 제약 조건을, 애플리케이션은 비즈니스 규칙을 각각 보장해야 한다.

2단계: 스펙트럼 선택

분산 환경에서 정합성은 공짜가 아니다. Strong에서 Eventual까지, 데이터 특성에 맞는 수준을 선택해야 한다.

3단계: 안전한 관리

Saga, Outbox, Event Sourcing, CQRS로 최종 정합성을 안전하게 관리한다. 밑바닥에는 멱등성이 있다.

"좋은 아키텍트는 '이 데이터에는 어느 수준의 정합성이 적절한가?'를 판단할 수 있는 사람이다."
